# 설계: 개발자 글쓰기 원칙 흡수 (wikidocs 20224 → CLEAR-COMM · domain-docs)

작성: 2026-06-16 · 확정

## 1. 배경과 목적

책 [글 잘 쓰고 싶은 개발자](https://wikidocs.net/book/20224)(wikidocs book 20224, 전 114편)를
크롤링해 `D:\projects\wikidocs_book_20224\`에 보존했다(raw + 마크다운 + INDEX). 책의 목적("글쓰기는
멋이 아니라 전달 — 오해를 줄인다")은 우리 `CLEAR-COMM`과 같은 영토다. 책을 distill해 그 *방법*(누구나
아는 "쉽게 써라"가 아니라 "추상→구체" 같은 how)을 디시플린에 흡수한다.

사용자가 정리한 세 목적은 *생성 시점의 표현·저작 방식*을 규정한다(사후 점검이 아니라 만들어내는 그 순간):
- ① **청자 기준 대화 표현력** → `CLEAR-COMM`. 매 답변을 생성하는 순간 어떻게 표현하느냐 — 같은 내용도
  표현에 따라 독자의 몰입과 피로도가 갈린다.
- ② **목적에 맞는 문서 양식** → `domain-docs`. 문서를 쓰는 시점에 이미 적절한 양식으로 짜고 필요한 내용을
  제자리(적절한 스타일)에 놓는다.
- ③ **산출물 객관 검증** → 리뷰어. 사후 검증이라 ①②와 완전히 별개 — 손대지 않는다.

플러그인 분해는 하지 않는다(모놀리식 유지) — 세 목적은 생성 시점 행동이지 플러그인 경계가 아니다.
앞선 분해 검토(2개 spec)는 YAGNI·동시성 비용으로 폐기했다.

## 2. 비목표

- 새 원칙 ID를 늘리지 않는다(2026-06-15 정책 — YAGNI·흡수). 7개 분석 테마는 뼈대일 뿐 `CLEAR-COMM`
  한 항목으로 흡수한다.
- 별도 "종합 참조 문서"를 만들지 않는다 — 풀 distill은 크롤링한 책이 한다(과설계 회피, 첫 3렌즈 리뷰의
  critical 반영).
- Part 13(매일 훈련)은 담지 않는다(사람 습관용).
- 플러그인 구조를 바꾸지 않는다.

## 3. 분석 뼈대 — 7개 테마 (배치는 4·5절)

13개 Part를 에이전트 소통 7개로 재클러스터. 우산 명제: "글쓰기는 전달 — 오해를 줄인다"(Part 01).
독자·결론 먼저(Part 05·06) / 쉽게=추상→구체(Part 02) / 짧게(Part 03·부록A) / 팩트·근거(Part 04·부록B) /
논리 뼈대(Part 10) / 리듬·가독성(Part 11) / 퇴고=삭제(Part 12).

배치: ①에 해당하는 보편 지시어 → `CLEAR-COMM`. ②에 해당하는 글 유형별 → `domain-docs`. 리듬·가독성
(Part 11)은 몰입·피로도를 좌우하므로 `CLEAR-COMM`에 포함한다(문장 변주·소제목 호흡) — 표현력이 곧 ①.

## 4. 변경 1 — `CLEAR-COMM` 보강 (① 대화 표현력)

`CLEAR-COMM`은 이미 결론 먼저·한 문장 한 개념·짧은 단락·용어 부연·완결 문장·읽는 사람 기준을 담는다.
책에서 **현재 문구에 없는 알맹이만** 같은 산문(완결 문장)으로 흡수한다. 새 IDs·불릿 나열 없음.

추가할 알맹이(정확한 diff):
- **추상→구체**(PLAIN): 모호·추상 표현을 구체 정보로 — 무엇이·언제·어떤 조건에서·얼마나·어떤 결과인지
  채운다("느리다" 대신 "로딩 12초"). 이 책의 핵심 *방법*.
- **팩트·근거**(FACTUAL): 의견과 사실을 구분하고, 확인 안 된 것은 단정 말고 "가능성/추가 확인"으로 표시,
  주장에는 확인 가능한 근거(로그·수치·재현조건). `EXPLICIT`의 소통판(코드↔글 영역이 달라 중복 아님).
- **논리 순서**(LOGICAL): 원인과 결과를 뒤섞지 않고, 일부 사례를 전체 결론으로 확대하지 않는다.
- **퇴고=삭제**(REVISE-BY-CUTTING): 다 쓴 뒤 목적과 안 맞는 문장·군더더기를 덜어낸다 — 고치기는 더하기보다
  덜어내기. 코드의 `SURGICAL`을 글에 적용(영역 달라 중복 아님).
- **리듬·표현력**(READABLE): 읽는 흐름을 살려 끝까지 몰입하게 한다 — 문장 길이를 단조롭게 반복하지 말고
  변주하며, 긴 답은 소제목으로 호흡을 끊는다. 같은 내용도 표현에 따라 피로도가 갈린다(① 핵심).

토큰 비용: 사용자가 수용(상시 한글 주입을 한국어 응답 품질 이득으로 봄).

## 5. 변경 2 — `domain-docs`에 "글 유형별 적용" (② 문서 양식)

`domain-docs`(이미 이 책을 README 출처로 distill 인용 중)에 개발자가 자주 쓰는 글 유형 체크리스트를 한
섹션 추가한다. 보편 원칙은 재서술 않고 ID로 참조(README 섹션과 동일 방침). 부록 C~G distill.

- 버그 리포트(부록 C): 제목=[환경/화면/현상]·재현 한 단계 한 행동·기대≠실제 분리·추측은 가능성으로.
- 코드 리뷰 코멘트(부록 D): 사람 아닌 코드·명령보다 이유·확신 낮으면 질문형·칭찬도 구체.
- 작업 보고(부록 E): 완료/진행/막힘/남은 일 분리·"테스트함"→환경·케이스 명시.
- 기술 블로그(부록 G): 요약(문제+결론) 먼저·환경 명시·해결 확인 기준·허세 빼기.
- 커밋·PR(도출, 책 밖): 무엇을·왜 먼저·추측/과장 배제(`CLEAR-COMM` 팩트·근거).

README는 domain-docs에 이미 전용 섹션이 있어 중복 추가 안 함.

## 6. SSOT·중복 점검 (FAIL-LOUD)

- `CLEAR-COMM`(소통) vs `domain-docs`(저작)는 영역이 다르다. domain-docs는 보편 알맹이를 ID 참조만.
- `FACTUAL`↔`EXPLICIT`, 퇴고=삭제↔`SURGICAL`은 글/코드로 분리 → 본문 한 줄로 관계 명시.
