# 설계: agent-principles 리디자인 (원칙 enrich · 절차 재구성 · 리뷰어 분리)

작성: 2026-06-15

## 1. 배경과 문제

disciplined-coder 플러그인은 팀 원칙(`coding-principles.md`)을 SessionStart 훅으로 모든 세션에 주입하고,
도메인 참고서와 런타임 검증 어드바이저를 스킬로 제공한다. 긴 사용 검토에서 다음 통증이 확인됐다.

1. **이름·범위가 좁다.** "코딩 디시플린"이지만 내용(소통·원본 보존·측정 등)은 코드 작성만이 아니라
   Claude Code의 모든 작업에 적용된다.
2. **원칙 항목이 너무 짧다.** ID만 던지고 목적어·동사를 생략해, 읽는 사람이 뜻을 복원해야 한다.
   이는 우리가 막 강화한 `CLEAR-COMM`("압축보다 명료함")을 원칙 파일 자신이 어기는 것이다.
3. **"절차" 섹션이 꼬여 있다.** "런타임 LLM 절차/계획 시점 절차"는 *언제*(트리거)로 이름 붙였는데
   "메타 산출물 리뷰 절차"는 *무엇*(출력)으로 이름 붙여 동질성이 없다. 검증 방법(PREP→렌즈→집계→라우팅)이
   메타 산출물 밑에만 적혀, 같은 방법을 쓰는 런타임 검증에서는 안 보인다.
4. **리뷰어 정의가 중복된다.** `domain-llm-runtime`의 리뷰어들(correctness/fit)과
   `advisor-spec-review`의 3렌즈(factual/consistency/adversarial)는 "무언가를 보고 문제 목록을 돌려주는
   리뷰어"라는 같은 추상을 두 곳에 적은 `SSOT` 위반이다.
5. **빈 스텁이 거짓 커버리지를 만든다.** `domain-ui/app/agent/db`는 범위 한 줄뿐인데 목차에 올라 있어
   "이 도메인은 다뤄졌다"는 착각을 준다(`FAIL-LOUD` 위반, 자기 YAGNI 원칙 위반).
6. **이슈 로그가 수동적이다.** solved_problems는 주입되지만 "꺼내 쓰라"는 트리거가 없어 사실상 안 쓰인다.

이 설계는 위 여섯을 한 번에 해소한다.

## 2. 비목표 (Non-goals)

- **검색형 이슈 로그**(항목별 파일 + description 기반 recall)는 만들지 않는다. 평면 리스트가 실제로
  커져서 아플 때 별도로 다룬다(측정 먼저).
- **SOLID·SE·카파시 ML 레시피를 담는 온디맨드 카탈로그 참고서**는 만들지 않는다. 쓸모 있는 알맹이는
  원칙 본문에 흡수한다.
- **어휘 복리 누적**(배운 용어를 기록해 다음엔 전제로 까는 기능)은 만들지 않는다.
- 새 원칙 ID를 늘리지 않는다. 추가 후보(에이전트 과설계 금지·직교성)는 기존 원칙에 흡수한다(YAGNI).

## 3. 원칙 파일 변경

### 3.1 파일명·제목·범위

- 파일명을 `coding-principles.md`에서 `agent-principles.md`로 바꾼다.
- 제목을 "코딩 디시플린 (팀 원칙)"에서 "디시플린 (팀 원칙)"으로 바꾼다("코딩" 한정 제거).
- 적용 범위 문구를 "모든 코드 작업에"에서 "모든 작업에"로 바꾼다.

### 3.2 우선순위 없음 → `NO-PRIORITY` 원칙 하나로

지금 "순서·우선순위는 없다"는 인트로 메타 문구로 흩어져 있다. 이를 원칙 용어집 안의 `NO-PRIORITY`
항목 **한 곳**으로 모으고, 인트로의 중복 문구는 지운다. 알파벳 순서상 `MEASURE-FIRST`와
`NON-DESTRUCTIVE` 사이에 둔다.

> **`NO-PRIORITY`** — 이 원칙들엔 우열·순서가 없다. ID로만 참조하는 알파벳순 용어집이며(번호·그룹 없음),
> 위아래 위치엔 의미가 없다. 상황에 맞는 원칙을 모두 적용한다.

### 3.3 외래어 정리

"글로서리"를 "용어집"으로 바꾼다.

### 3.4 모든 원칙 enrich

12개 원칙(+`DOCKER-ONLY`)을 ID만 던지는 형태에서, ID를 풀어 쓰고 목적어·동사를 갖춘 완결된 문장으로
충분히 설명하는 형태로 올린다. 구루 출처는 **표기하지 않는다**(AI 행동을 제약하지 않고 토큰만 먹으며,
링크를 따라가 볼 일도 없으므로). 대신 그 조언의 알맹이를 문장에 녹인다. 목표 스타일은 아래 샘플과 같다.

> **`SSOT` (Single Source of Truth · 단일 출처)** — 하나의 사실·설정·결정은 시스템에서 한 곳에만
> 권위 있게 둔다. 다른 데서 필요하면 복제하지 말고 그 한 곳을 참조하거나 거기서 도출한다. 사람이 두 곳을
> 손으로 맞추는 순간 둘은 반드시 어긋나기 때문이다.

> **`FAIL-LOUD` (조용한 실패 금지)** — 무언가 틀어지면 즉시, 눈에 띄게 터뜨린다. 계약 위반이나
> 드리프트(설정이 코드와 어긋남)를 삼키고 그냥 진행하지 않는다. 구조(불변식·생성된 설정·명확한 계약)가
> 실수를 막거나 드러내게 만들어, "작성자가 정확히 기억하기"에 의존하지 않는다.

### 3.5 새 원칙 `SURGICAL`

AI 에이전트가 시키지도 않은 주변 코드를 건드리는 통증을 막는 원칙을 추가한다(카파시 "Surgical Changes"
조언의 알맹이).

> **`SURGICAL` (꼭 필요한 곳만)** — 요청과 직접 연결된 줄만 바꾼다. 작동하는 주변 코드를 리팩터·정리·
> 재포맷하지 않는다. 죽은 코드는 삭제하지 말고 표시(flag)하며, 내 변경으로 불필요해진 것만 제거한다.
> 주변 스타일을 따른다.

### 3.6 `CLEAR-COMM`에 소통 규약 박기 (이미 1차 반영, 마저 확정)

`CLEAR-COMM`은 "어떻게 소통하나"의 단일 출처다. 다음을 모두 담는다(일부는 이미 반영됨).

- 결론을 먼저, 근거·원리는 뒤에 둬 읽다 멈춰도 되게 한다(점진적 공개).
- 한 문장에 한 개념, 단락은 짧게.
- 전문영역 용어는 나올 때마다 짧게 부연한다(독자가 그 분야를 안다고 가정하지 말 것).
- 짧은 답보다 피로도 낮은 답을 택한다(토큰 아끼려 압축하지 않는다).
- 단어나 구만 늘어놓는 방식으로 쓰지 말고, 목적어와 동사를 갖춘 완결된 문장으로 풀어 쓴다.
  화살표나 슬래시를 문장 안에서 보조로 쓰는 것은 괜찮다.

### 3.7 흡수 (새 ID 없이)

- **에이전트 과설계 금지**(Anthropic: 단일 LLM 콜로 충분하면 에이전트로 키우지 마라, 지연·비용 트레이드오프
  고려)를 `SIMPLE`에 흡수한다.
- **직교성**(관련 없는 것들은 독립적으로, 한 곳을 바꿔도 다른 곳이 안 깨지게)을 `FOCUSED`에 흡수한다.

### 3.8 공통 함정 enrich

`.gitignore` 인라인 주석과 mock/None 항목을 짧은 단편에서 "왜 그런지"까지 설명하는 문장으로 올린다.
"매직 넘버 금지" 항목은 이미 명료하므로 그대로 둔다.

## 4. "절차" 섹션 재구성

세 종류(검증 / 설계 입력 / 장부)를 섞고 제목이 들쭉날쭉한 현 구조를, 종류별로 묶고 공통 방법을 위로
끌어올린 구조로 바꾼다.

### 가. 검증 레이어 — 단독 LLM 출력으로 끝내지 마라

공통 규칙: LLM이 만든 결과는 — 제품 출력이든 Claude의 설계 문서든 — 단독 결론으로 마치지 말고
리스크에 비례해 리뷰를 더한다. 어느 리뷰어를 쓸지는 트리거가 정한다.

| 트리거 (언제) | 검증 대상 | 호출자 | 강제 방식 |
|---|---|---|---|
| 제품이 런타임에 LLM 호출 | 제품의 LLM 출력 | `domain-llm-runtime` | 제품 코드로 구현 |
| spec 작성 | Claude의 설계 문서 | `domain-spec-review` | 훅이 강제 |
| plan 작성 | Claude의 계획 문서 | `domain-spec-review` | 훅이 강제 |

공통 방법(여기 한 번만, 호출자들이 참조)을 완결된 문장으로 적는다.

검증은 네 단계로 진행한다. 먼저 **PREP**으로, 무엇을 검증할지 미리 정하고 필요한 배경 지식을 함께
준비한다(즉흥 금지 — TDD에서 테스트를 먼저 쓰는 것과 같은 발상). 그다음 **독립 렌즈**로, 서로 다른
관점의 리뷰어 여럿이 동시에 각자 따로 본다(작성자 편향 회피, 읽기 전용). 그다음 **메타 집계**로,
리뷰어들의 지적을 모아 상충과 빈틈을 점검한다(내용 재판단은 하지 않는다). 마지막으로 **라우팅**으로,
결과에 따라 통과(accept)·재생성(regenerate)·사람에게 올림(escalate) 중 하나를 고른다.

핵심 정리: 메타 산출물 리뷰는 별개의 낯선 것이 아니라, 이 검증 방법을 *Claude 자신의 설계 문서*에
적용한 경우다. 런타임 어드바이저는 같은 방법을 *제품 출력*에 적용한 경우다. 방법은 같고 대상만 다르다.

### 나. 설계 입력 — domains-index 확인

설계/계획 시, 만들 대상이 도메인 참고서에 있으면 열어 "마땅히 그래야 하는 것"을 명세에 반영한다
(명세에 못 담으면 개발 단계에서 고려). 이는 검증이 아니라 요구사항 수집이라 따로 둔다.

### 다. 이슈 로그 — solved/unsolved 운영 (생애주기 + 작성자 통합)

서브에이전트도 이 글을 읽으므로, 읽는 주체가 자기 역할을 바로 알게 쓴다.

- **등록**: 검증/리뷰가 끝나 문제를 인지하면 `unsolved_problems.md`에 적는다. 형식은 "증상/트리거 →
  교훈(다음엔 이렇게)"으로, 서술이 아니라 처방이 앞에 오게 한다.
- **이동**: 테스트가 통과로 바뀌면 `solved_problems.md`로 옮긴다.
- **recall(꺼내 쓰기)**: 디버깅이나 구현을 시작하기 전에 solved에서 유사 증상을 먼저 찾는다.
- **작성자**: 이 파일은 메인 세션만 쓴다(동시 쓰기 손상 방지). 네가 서브에이전트라면 로그를 직접
  쓰지 말고, 발견한 문제를 리턴값으로 보고하라. 메인이 취합·dedup해 기록한다.
- **🔴 금지**: `unsolved_problems.md`의 🔴(사용자 결정 필요)는 누구도 자율 구현하지 않는다.

## 5. 리뷰 아키텍처 재배치 (리뷰어 분리)

### 5.1 설계도의 두 종류 (라벨)

스킬 문서는 "돌아가는 것"이 아니라 "설계도(blueprint)"다. 설계도는 두 종류이며, 문서에 명시한다.

- **프롬프트 설계도** — LLM에게 먹이는 리뷰어. `reviewer-*`.
- **코드 설계도** — 결정론적 코드로 구현되는 것. `meta-aggregate`(집계·결정)와 비기능 체크리스트.

### 5.2 리뷰어 풀 (프롬프트 설계도, 렌즈만 담음)

각 리뷰어는 렌즈 하나(무엇을 보고 어떤 문제 목록 JSON을 돌려주는가)를 정의하며, 어떻게 실행되는지는
신경 쓰지 않는다. 같은 렌즈를 두 호출자가 각자 다른 방식으로 감싼다.

| 리뷰어 | 렌즈 | 주 사용 호출자 |
|---|---|---|
| `reviewer-grounding` | 출력·주장이 그 출처(요청+맥락, 또는 외부 사실)에 충실한가 — 누락·모순·환각 | 둘 다 |
| `reviewer-fit` | 출력이 소비자 계약(형식·스키마·스타일·제약)을 지키는가 | 런타임 |
| `reviewer-consistency` | 내부 모순, spec↔plan 커버리지 공백, 이름·타입 드리프트 | spec 리뷰 |
| `reviewer-adversarial` | 실패 모드, 과설계, YAGNI 위반, 비가역성 | spec 리뷰 |

`reviewer-grounding`은 기존 `advisor-correctness`(런타임)와 spec 3렌즈의 factual을 하나로 합친 것이다
(둘 다 "주장이 출처에 근거하는가"를 본다 — `SSOT` 극대화). 출처가 무엇인지는 호출자가 준다. 런타임에서는
원래 요청과 제공된 맥락이 출처이고, spec/plan 리뷰에서는 검토 대상 문서와 PREP으로 주입된 선행 결정·
검증할 구체 사실이 출처다. 네 리뷰어는 각각 자기 `skills/reviewer-*/SKILL.md` 파일을 갖는다
(`reviewer-consistency`와 `reviewer-adversarial`는 기존 `advisor-spec-review` 본문에서 추출한다).

### 5.3 meta-aggregate (코드 설계도, 두 호출자 공유)

리뷰어가 둘 이상일 때 그 결과를 모아 다음 행동을 정하는 집계·결정 단계다. 리뷰어가 아니다(렌즈가 아님).

- **하는 일**: 지적을 한 목록으로 모아 출처 태깅·심각도 정렬, 같은 지점의 상반 판정(상충) 감지,
  아무도 안 본 차원(공백) 감지, 그리고 결정(critical≥1이면 regenerate, 상충/공백이면 escalate,
  깨끗하면 accept).
- **하지 않는 일**: 내용 재판단. "리뷰어 A가 옳다"를 정하지 않는다(무한 재귀 회피 — 끝은 사람).
- **구현 형태(맥락 의존)**:
  - 제품 런타임에서는 결정론적 파이썬 함수로 구현한다(LLM 콜 아님). 집계·계수는 결정론이므로 모델이 필요 없다.
  - spec/plan 리뷰에서는 제품 코드가 없으므로 메인 세션(Claude)이 좁은 절차를 따라 직접 집계한다.
    스크립트를 싣지 않는 이유는, 이 플러그인의 훅이 의도적으로 순수 bash(jq·python 비의존)로 짜여
    이식성을 지키기 때문이다(대상 PC에 python이 없을 수 있음). 단순한 집계라 좁은 절차로 충분하며,
    강한 결정론이 필요한 제품 런타임 쪽은 진짜 코드로 이미 보장된다.

### 5.4 비기능 체크리스트 (코드 설계도, 런타임 전용)

런타임에 LLM/외부 서비스를 부르는 코드가 갖춰야 할 요건이다(타임아웃·재시도·None 가드·에러 형식·비용
상한·관측·HITL·민감정보). 리뷰어가 아니고 LLM 콜도 아니다. 구현하는 순간 호출 코드의 가드와 그것을
검증하는 테스트로 확정된다. spec/plan 리뷰에는 제품 코드가 없어 적용되지 않으므로, 런타임 호출자
`domain-llm-runtime` 안에 둔다(공유 대상이 아님).

### 5.5 호출자 둘

- **`domain-llm-runtime`** (런타임): 리스크 점수로 리뷰어를 고른다(0–1: 비기능만 · 2–3: +grounding ·
  4–5: +fit +meta). 고른 리뷰어를 제품 코드의 리뷰 콜로 구현하고, 비기능 체크리스트는 항상 적용하며,
  meta는 결정론 함수로 둔다.
- **`domain-spec-review`** (spec/plan 리뷰, `advisor-spec-review`에서 이름 변경): Claude가 spec/plan을
  쓰면 grounding·consistency·adversarial 리뷰어를 각각 **별도의 읽기 전용 Claude Code 서브에이전트**로
  띄우고(리뷰어당 하나씩 — 한 에이전트가 모든 렌즈를 몰아 보는 것을 구조로 막음), meta로 집계·라우팅한다.
  훅이 강제하며(PostToolUse 감지 + Stop 게이트), 완료 시 문서 마지막 줄에 `spec-review` 마커를 남긴다.

### 5.6 이전 결정 폐기 (정직히)

지난 논의의 "advisor 4종을 `domain-llm-runtime`의 references로 흡수" 결정은 폐기한다. 리뷰어들이 이제
두 호출자가 공유하므로, 한 호출자 안에 넣으면 다른 호출자가 못 본다. 따라서 리뷰어는 한 단계 위,
둘 다 참조할 수 있는 자리(`reviewer-*`)에 둔다. 더 나은 `SSOT` 결과다.

`coding-principles.md` → `agent-principles.md` 변경을 다음 살아있는 배선에 반영한다(과거 docs/superpowers
