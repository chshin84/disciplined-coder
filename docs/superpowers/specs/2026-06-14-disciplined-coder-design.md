# disciplined-coder 설계 문서

**날짜**: 2026-06-14
**상태**: 설계 확정 (구현 계획 대기)
**대상**: 솔로 사용자(chshin84), 모든 프로젝트에 적용

---

## 1. 목적

"규율 있는 코딩"을 모든 프로젝트·모든 에이전트에 일관되게 강제하기 위한 단일 Claude Code 플러그인. 세 가지 서브시스템으로 구성되며, 셋 다 본질적으로 **"원칙을 세우고 강제한다"**는 한 줄기다.

1. **디시플린(원칙)** — 항상 주입되는 린한 엔지니어링 원칙 세트
2. **어드바이저 시스템** — 제품이 **런타임에 LLM을 호출할 때** 붙일 검증 레이어의 구현 스펙
3. **이슈 로그** — 프로젝트별 solved/unsolved 문제의 생애주기 관리

---

## 2. 스코프

### 포함
- 솔로 사용 전제. user scope 설치 → 사용자의 모든 프로젝트에 자동 적용.
- 디시플린은 모든 커스텀 서브에이전트에 자동 도달(빌트인 Explore/Plan 제외 — 무관하다고 사용자 확인).
- 어드바이저 시스템의 **포커스는 "런타임 LLM 동작"** (제품 코드에 구현할 검증 패턴).
- 이슈 로그 생애주기: 자동(체크포인트 연동).

### 제외
- **개발시점 리뷰 오케스트레이션은 만들지 않는다.** superpowers 등이 이미 개발시점에 리뷰어를 붙여 검증하므로 중복.
- CleanHarness 참조/의존 없음(CleanHarness는 삭제 예정). 단, 그 안의 리뷰어/메타 정의의 **가치를 어드바이저 스펙으로 증류해 보존**한다.
- 팀 마켓플레이스 배포는 현재 목표 아님(필요 시 후속 재패키징). GitHub repo는 정본 보관용으로 유지.

## 3. 핵심 메커니즘 사실 (설계의 근거, 공식 문서 검증 완료)

설계가 이 사실들에 의존하므로 명시한다.

1. **서브에이전트는 시작 시 CLAUDE.md 계층 전체를 로드한다**(프로젝트 CLAUDE.md + `@import` 포함). 예외는 빌트인 **Explore/Plan** 둘뿐. → 프로젝트 CLAUDE.md에 넣은 것은 모든 커스텀 서브에이전트에 자동 도달.
2. **`@import`은 세션 시작 시점의 스냅샷**이다. 세션 중 파일 변경은 이미 로드된 컨텍스트에 자동 반영되지 않음(`/compact` 시 루트 CLAUDE.md 재로드). 새로 뜨는 서브에이전트는 spawn 시점에 디스크에서 새로 읽음.
3. **플러그인 자체 CLAUDE.md는 컨텍스트에 로드되지 않는다.** 메인 주입은 skill / 프로젝트 CLAUDE.md `@import` / hook stdout으로 한다.
4. **SessionStart hook은 호스트 셸에서 실행**된다(컨테이너 아님). Windows는 Git Bash 필요(없으면 PowerShell 폴백으로 bash 실패). hook stdout은 메인 세션 컨텍스트로 주입된다.
5. **superpowers brainstorming/writing-plans 스킬은 메인 세션에서 실행**되므로 CLAUDE.md를 본다. 단 **빌트인 Plan 서브에이전트는 CLAUDE.md를 건너뛴다**(raw Plan에 위임 시 dispatch 주입 필요).
6. **배포된 제품의 런타임엔 Claude Code 서브에이전트가 없다.** 런타임 검증은 제품 코드가 직접 다중 LLM 콜로 구현한다 → 어드바이저는 "동작하는 에이전트"가 아니라 "구현 스펙".

## 4. 아키텍처 개요

```
disciplined-coder 플러그인 (user scope 설치 → 전 프로젝트)
├─ ① 디시플린 (항상 주입, 린)
├─ ② 어드바이저 시스템 (런타임 LLM 검증 패턴 구현 스펙)
│    ├─ 인덱스 (경량, 항상 노출)
│    └─ 스펙 스킬 4종 (온디맨드)
├─ ③ 이슈 로그 (solved/unsolved 생애주기)
└─ 전달: SessionStart hook이 프로젝트 CLAUDE.md에 디시플린+인덱스+이슈로그 @import를 멱등 배선
         (+ 첫 세션 구멍 보강용 stdout 주입). 디시플린/인덱스 정본은 플러그인에 1벌(SSOT).
```

## 5. 서브시스템 ① 디시플린 (원칙)

### 5.1 원칙 세트 (Tier 1, 항상 주입, 한 줄씩 — 12개 + 환경관례)

18개에서 병합·이동으로 12개로 정리했다. 이동분(구조적 분리·판단 재귀 회피)은 §6 어드바이저 시스템의 근거로, 경계+버전링은 Tier 2 변경관리 참조로 내렸다(§5.5).

> **A. 협업·커뮤니케이션**
> 1. **명료한 소통 + 판단 유예 금지** — 결론만 던지지 말고 근거·기각한 대안까지 설명한다. 단계를 건너뛰지 않는다. 판단이 필요한 곳에서 모호한 추측 대신 명확히 질문한다.
>
> **B. 설계·유지보수**
> 2. **단순·가독 (YAGNI)** — 지금 필요 없는 일반화·추상화 금지, 가장 단순한 동작부터. 코드는 쓰기보다 읽히는 횟수가 많다 — 주변 관례를 따른다.
> 3. **명시성 > 암묵성** — 놀라움 최소화. 숨은 동작·매직 금지. 의도는 이름·계약·타입으로 드러낸다.
> 4. **작은 단위·단일 책임** — 한 단위는 한 가지 일. 내부를 몰라도 쓸 수 있게 인터페이스로 의존. 파일 비대화는 분리 신호.
>
> **C. 정확성·안전**
> 5. **SSOT + 복제 말고 도출** — 같은 정보는 한 곳에만. 사람이 두 곳을 동기화하게 만들지 않는다.
> 6. **Fail-loud + 강건성 > 작성자 정확성** — 조용한 실패 금지(드리프트·계약 위반은 즉시 빨강으로). 구조(불변식·생성된 설정·명확한 계약)가 실수를 막거나 드러내게 한다 — "정확히 기억"에 의존 금지.
> 7. **TDD + 검증 후 주장** — 실패 테스트 먼저. 실행 증거 없이 "됐다" 금지.
> 8. **비파괴 우선** — 원본 보존. 필터/dedup/스팸 제거는 삭제가 아니라 분류 표시(mark). 저장 후 처리.
> 9. **측정 먼저** — 가정 금지. 환경 차이를 먼저 확인.
> 10. **멱등성** — 스크립트·마이그레이션·셋업은 반복 실행해도 안전하게.
> 11. **비밀 분리** — 진짜 비밀은 백엔드 전용. 비밀 아닌 식별자만 클라이언트 노출.
> 12. **가역성 우선** — 되돌릴 수 있는 결정을 선호. 비가역 결정은 근거를 남긴다.
>
> **환경 관례 (보편 원칙 아님, 명시)**
> · **Docker 전용 개발/테스트** — 로컬 실행 금지. 프로덕션과 동일 환경에서.

### 5.2 절차 지시 (원칙과 별개, 어드바이저·계획 연결)

디시플린 안에 두는 짧은 연결 두 줄:
- **런타임 LLM 절차**: "LLM 콜로 끝나는 런타임 기능은 단독 결론으로 마치지 말고, 어드바이저 인덱스에서 적합한 검증 레이어를 골라 제품 코드에 구현하라."
- **계획 시점 절차**: "설계/계획(brainstorming/writing-plans) 시 런타임 LLM 기능은 어드바이저 검증 레이어를 계획에 반드시 포함하라. 포함 여부·범위는 인덱스의 리스크 기준으로 선택."

### 5.3 전달
- 디시플린 정본 = 플러그인 내 `coding-principles.md` (1벌, SSOT).
- SessionStart hook이 프로젝트로 복사 + 프로젝트 CLAUDE.md에 `@import` 배선 → 메인 + 모든 서브에이전트 도달.
- 첫 세션 `@import` 타이밍 구멍은 hook stdout으로 디시플린 다이제스트를 출력해 보강.

### 5.4 planning 영향 (사용자 요청 B)
5.2의 "계획 시점 절차" + §6.5의 어드바이저 인덱스가 CLAUDE.md(항상 로드)에 있으므로, 메인 세션에서 도는 superpowers brainstorming/writing-plans가 이를 보고 런타임 검증 레이어를 계획에 반영한다. superpowers 파일은 수정하지 않으며 오직 컨텍스트로만 영향을 준다(§3-5 근거).

### 5.5 Tier 2로 내린 항목 (변경관리)
- **불변성보다 경계 + 버전링** — 변경을 금지하지 말고 언제(작업 경계)·누가(단일 권한)·어떻게(버전·로그)를 통제한다. 항상 주입하지 않고, 명세/규칙 변경을 다루는 작업에서 온디맨드 참조(또는 §7 이슈 로그 규약과 함께)한다.

## 6. 서브시스템 ② 어드바이저 시스템 (런타임 LLM 검증 패턴)

### 6.1 정체
배포 제품이 런타임에 LLM을 호출할 때, **단독 워커 콜로 끝내지 않고 독립 검증 레이어를 붙이는 패턴의 구현 스펙**. Claude Code 에이전트가 아니라, 제품 코드가 다중 LLM 콜로 구현할 **청사진**이다(§3-6). 따라서 `agents/advisor-*.md`(동작하는 서브에이전트 정의)는 만들지 않는다.

이 시스템의 설계 근거는 §5.1에서 이리로 옮긴 두 원리다: **전능 에이전트보다 구조적 분리**(중요한 LLM 작업을 단일 콜에 몰지 말고 역할/차원을 분리해 품질 확보), **판단 재귀 회피**(상위 메타 레이어는 내용을 재판단하지 않고 구조적 건강성만 점검 — 재귀의 종단은 사람). 이 둘은 어드바이저를 쓸 때만 의미가 있으므로 항상 주입하지 않고 여기 둔다.

### 6.2 4종 구성 (CleanHarness에서 증류)

| 어드바이저 | 렌즈 |
|---|---|
| 정합성 (A) | 명세/요구 대비 누락·불일치 |
| 비기능 (B) | retry / timeout / HITL / 에러 응답 형식 / 관측 |
| 적합성 (C) | 원칙·스타일·재사용 적합성 |
| 메타 (Meta) | A·B·C 출력의 구조적 건강성(상충·커버리지 공백·품질저하)만. **내용 재판단 금지**(판단 재귀 회피 원리, §6.1) |

### 6.3 조립 (런타임 흐름)
- 정합성/비기능/적합성 3종은 **병렬 LLM 콜**.
- 메타는 그 3종 출력을 받아 **뒤에 도는 2단계**(병렬 아님 — 판단 재귀 회피).
- 결과를 제품이 취합(severity 정렬·그룹핑) → critical 처리 루프.

### 6.4 리스크별 선택 로직 (인덱스에 수록)
체크리스트 점수로 포함 범위 결정(잠정값, CleanHarness 기준 차용):

| 체크 항목 | 점수 |
|---|---|
| 외부 서비스/API 호출 포함 | +1 |
| LLM 컴포넌트 포함 | +1 |
| 인터페이스 계약 변경 | +1 |
| HITL/컴플라이언스 접점 | +1 |
| 명세 3개 이상 섹션 관여 | +1 |

| 점수 | 구성 |
|---|---|
| 0–1 | 비기능(B)만 |
| 2–3 | 정합성(A) + 비기능(B) |
| 4–5 | A+B+C + 메타 |

### 6.5 산출물 구조
- **인덱스**(경량, 디시플린에 노출): 4종 목록 + 한 줄 성격 + 6.4 선택 로직 + 6.3 조립도. → 플래너가 "언제/무엇을" 선택하는 근거.
- **스펙 스킬 4종**(온디맨드, `skills/advisor-<name>/SKILL.md`): 각 어드바이저별로
  - 렌즈/체크리스트
  - 레퍼런스 프롬프트 템플릿(system/user)
  - 출력 스키마(severity 태깅 이슈; JSON 권장)
  - 배선 노트(병렬/순차, 메타 후행, 취합·재호출 루프, 비용·지연 고려)
- 복붙해서 제품 코드로 옮길 수 있는 수준으로 작성하되, 플래너가 인덱스로 적절히 선택한다(사용자 요청 2).

## 7. 서브시스템 ③ 이슈 로그 (solved/unsolved 생애주기)

### 7.1 파일
- `solved_problems.md` — 해결된 문제(문제→원인→해결).
- `unsolved_problems.md` — 미해결/대기. 범례 🔴 결정 필요(자율 구현 금지) / 🟡 구현 대기 / 🔵 향후.
- SessionStart hook이 없으면 생성(멱등) + 프로젝트 CLAUDE.md `@import` 배선.

### 7.2 생애주기 (B: 자동 체크포인트 연동 + D: 조합)
- **서브에이전트는 문제를 "보고"만** 한다(구조화된 리턴값). 파일 직접 쓰기 금지.
- **메인 세션(오케스트레이터)이 단일 작성자.** 서브들의 리턴을 취합·dedup 후 기록(동시 쓰기 손상 방지).
- **체크포인트(D)**:
  - **등록**: 검증/리뷰 작업(예: "검증해줘", "문제 찾아줘", /code-review) 종료 시 → unsolved에 반영.
  - **solved 이동**: 테스트가 통과로 전환될 때(코더가 docker 테스트 green). PostToolUse hook로 보조 가능.
- 이 체크포인트는 의미 판단이라 hook 자동 감지 불가 → **디시플린 지시(모델 주도)**로 박는다.

### 7.3 세션 중 갱신 대응
- 메인: 자기가 작성하므로 항상 최신 인지.
- 새로 뜨는 서브: spawn 시 디스크에서 새로 로드 → 갱신 후 최신 반영.
- 확실성 보장: 메인이 서브 dispatch 시 **관련 현재 항목을 프롬프트에 주입**(@import 재로드 타이밍 무관).
- 동시 실행 서브끼리는 실시간 공유 불필요 — 리턴 후 메인이 취합.

### 7.4 🔴 자율구현 금지 전파
- `unsolved_problems.md` 상단에 "모든 에이전트는 🔴 항목을 자율 구현하지 마라"는 강한 명령을 둔다 → @import로 모든 서브에이전트에 함께 전달.
- CLAUDE.md는 강제가 아닌 가이드이므로, 진짜 차단이 필요하면 PreToolUse hook로 강제(후속 옵션).

## 9. 전달·설치

- `claude plugin install disciplined-coder --scope user` → 모든 프로젝트에서 hook 동작.
- **Windows 전제**: 호스트에 Git Bash 필요(§3-4). README 상단 Prerequisites에 명시.
- hook은 `bash "...scaffold.sh"`로 호출(실행권한 비트 의존 없음).

---

## 10. 미해결 / 후속 결정

1. **디시플린 분량**: 18→12개(+환경관례)로 정리 완료(§5.1). 추가 trim 필요 시 리뷰에서 결정.
2. **어드바이저 스펙 상세도**: "복붙 구현 가능" 수준의 깊이를 어디까지 — 레퍼런스 프롬프트·스키마를 언어별로 둘지(우선 언어 중립 + 의사코드).
3. **🔴 PreToolUse 강제**: 가이드를 넘어 하드 차단을 둘지(후속).
4. **이슈 로그 체크포인트의 결정론 보조**: 테스트 통과 PostToolUse hook 패턴(테스트 명령 식별) 구체화.
5. **첫 세션 stdout 주입 포맷**: 다이제스트 길이/구조.

## 11. 검증 방법

- `scaffold.sh`: bash 문법 검사 + 신규/기존 프로젝트 멱등성 + 산문 충돌(@import whole-line) 회귀 테스트(이미 통과 이력 있음).
- 어드바이저 인덱스가 CLAUDE.md @import 경로로 메인 세션에 실제 로드되는지 확인(`/memory`).
- 디시플린이 일반 서브에이전트 dispatch 시 도달하는지 샘플 확인.
- 플러그인 검증: `claude plugin validate ./disciplined-coder --strict`.

## 12. 구현 단계 (phasing)

세 서브시스템을 한 번에 만들지 않고 단계로 나눈다. 각 단계는 **독립적으로 검증 가능**하며, 다음 단계는 이전 단계의 토대를 재사용한다.

### Phase 1 — 토대 + 디시플린 (먼저)
- §5.1 정리된 원칙 12개(+Docker)를 `coding-principles.md`(SSOT)로 작성.
- `scaffold.sh`(멱등) 작성: principles를 프로젝트로 복사 + 프로젝트 CLAUDE.md에 `@import` 배선 + 첫 세션 stdout 주입.
- `hooks/hooks.json`(SessionStart, `bash` 호출), `plugin.json`, README Prerequisites(Git Bash).
- user scope 설치 + `--strict` 검증.
- **검증**: 디시플린이 메인 + 일반 서브에이전트에 실제 도달하는지(`/memory`, 샘플 dispatch).
- **이유**: 전달 메커니즘(hook→CLAUDE.md)이 Phase 2·3의 토대다. 가장 작고 ROI 높음.

### Phase 2 — 어드바이저 시스템 (런타임 포커스, 사용자 주 관심)
- `advisors-index.md`(경량): 4종 + 성격 + §6.4 리스크 선택 + §6.3 조립도.
- `skills/advisor-{correctness,nonfunctional,fit,meta}/SKILL.md` 4종: 렌즈/체크리스트 + 레퍼런스 프롬프트 + 출력 스키마 + 배선.
- 디시플린에 절차 2줄(§5.2) 추가 + 인덱스를 CLAUDE.md 주입 경로에 연결(planning 영향).
- **검증**: 인덱스가 메인 세션에 로드되어 플래너가 선택 근거로 쓰는지. 스펙이 복붙 구현 가능한 수준인지.

### Phase 3 — 이슈 로그
- `scaffold.sh` 확장: `solved_problems.md`/`unsolved_problems.md` 생성(멱등) + `@import` 배선 + 🔴 자율구현 금지 명령.
- 디시플린에 생애주기 지시(B+D: 검증/리뷰 시 등록, 테스트 통과 시 solved 이동, 단일 작성자, dispatch 주입) 추가.
- (선택) 테스트 통과 PostToolUse hook 보조.
- **검증**: 멱등성 + 산문 충돌 회귀 + 🔴 명령 전파.

### 순서 근거
Phase 1은 전달 토대라 반드시 선행. 이후 사용자 주 관심인 **Phase 2(어드바이저)**를 먼저, 그다음 **Phase 3(이슈 로그)**. 단 2와 3은 서로 독립이라 순서 교체 가능.
